# WebAR · Складская навигация — PoC

Прототип по ТЗ v2.0. Веб-приложение без установки: наводите камеру телефона
на QR-код или штрихкод ячейки — поверх метки появляется табло с остатком и
статусом.

## Что реализовано

| ТЗ | Реализация |
|---|---|
| п.2, User Story | Полный сценарий: наведение → распознавание → табло → fallback на ручной ввод через 3с |
| п.3.1, Технологический стек | ZXing (`@zxing/library`) — один движок для QR и линейных штрихкодов (Code128/EAN), работает в Safari iOS и Chrome Android |
| п.3.2, UI/UX | AR-табло с цветовым кодированием (зелёный/красный/жёлтый), индикатор загрузки |
| п.3.3, Данные | Клиент `Api.getCell()` с кэшированием в `localStorage` и пометкой "неактуально с HH:MM" при потере сети |
| п.4, Устройства | `facingMode: environment`, пауза камеры при уходе вкладки в фон |
| п.6, Безопасность | Место под Bearer-токен в `api.js`, HTTPS обязателен для доступа к камере |

**Middleware API ещё не готов** (см. открытые вопросы в ТЗ) — используется
`js/mock-api.js`, имитирующий будущий эндпоинт `GET /api/v1/cell/{marker_id}`
со случайной задержкой и ~5% шансом сетевой ошибки (чтобы сразу проверить
поведение при плохой связи). Когда появится реальный API, правится только
`fetchCellFromNetwork()` в `js/api.js` — интерфейс ответа не меняется.

```
    └── qr-DEMO-00*.png     # тестовые QR-метки для демо (см. ниже)
```

## Запуск

Камера в браузере требует **HTTPS** (или `localhost`). Открыть `index.html`
через `file://` не получится — нужен любой статический сервер:

```bash
cd webar
npx serve .            # или: python3 -m http.server 8443 --bind 0.0.0.0
```

Для теста на реальном телефоне в локальной сети нужен HTTPS-туннель, например:

```bash
npx localtunnel --port 3000
# или ngrok http 3000
```

Откройте полученный `https://` URL на телефоне.

## Тестовые метки

В `assets/` — три готовых QR-кода (`DEMO-001`…`DEMO-003`), которым в
`mock-api.js` соответствуют три статуса (свободна / резерв / заблокирована).
Распечатайте их на матовой бумаге — по ТЗ п.5 минимум 10×10 см для
считывания с 1–1.5 метров (файлы уже в высоком разрешении для печати).

Для тестирования с реальными штрихкодами склада правка не требуется:
приложение декодирует то, что закодировано в метке, и ищет это значение
как `marker_id`. Единственное условие — ID в метке должен совпадать с
ключом записи в системе (сейчас — в `MOCK_DB`, затем — в реальной WMS).

## Что нужно проверить на реальном устройстве (не проверяется кодом)

Эти пункты из критериев приёмки (ТЗ п.7) — про физику и требуют теста на
месте, а не программной проверки:

- угол считывания до 45°;
- освещённость от 100 лк;
- автономность ≥4 часов непрерывной работы;
- фактическая задержка сети на Wi-Fi склада (сейчас эмулируется 250–900мс).

## Картинки товаров

Данные о ячейках реальные — выгрузка `65_ряд_1_5.xls` (91 ячейка, поля
`cell/sku/name/qty/status` в `js/mock-api.js`).

Картинки товаров лежат **локально** в `assets/photos/`, по имени файла =
артикул (SKU): `assets/photos/995417.jpg`. Это ровно тот формат, в котором
`fora_parser.py` уже сохраняет фото в свою папку `photos/` — можно просто
скопировать её содержимое в `assets/photos/` этого проекта, ничего не
переименовывая.

Поддерживаются `.jpg / .jpeg / .png / .webp` — `js/image-api.js` сам
перебирает расширения. Если фото для SKU нет — карточка ячейки просто
показывается без превью, ничего не ломается.

Так как файлы статические и лежат на том же домене (GitHub Pages), внешний
API/прокси для картинок не нужен — в отличие от прямого обращения к
api.catalog.ecom.fora.ua, которое браузер заблокирует из-за CORS.

*(В `worker/image-proxy.js` оставлен вариант с live-подгрузкой картинок
прямо с fora.ua через Cloudflare Worker — на случай, если понадобится
не хранить фото в репозитории, а тянуть их каждый раз актуальными.
Для текущего подхода с локальными файлами он не требуется.)*

## Известные ограничения PoC

- `mock-api.js` — временная заглушка, см. открытые вопросы в ТЗ (способ
  подключения к 1С ещё не определён).
- Позиционирование табло — по центроиду метки в 2D (через `object-fit: cover`
  пересчёт в экранные координаты), без полноценного 3D-трекинга плоскости.
  Для PoC этого достаточно; если потребуется устойчивое позиционирование
  при движении камеры — следующий шаг: WebXR Image Tracking с фолбэком на
  текущую реализацию для Safari.
- Bearer-токен на PoC захардкожен (`poc-demo-token`) — реальная выдача
  токена на смену не входит в объём PoC.
